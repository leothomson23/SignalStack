===============================================================================
 SignalStack v1.0.0 — Vulnerability Reference
 Difficulty: HARD
 For authorized security testing and education only.
===============================================================================

DESIGN PHILOSOPHY:
  This application is deliberately MORE SECURE than ShopZone and NexusBoard.
  It uses ORM everywhere (no raw SQL), proper password hashing (bcrypt, cost 12),
  JWT with algorithm restriction (HS256 only), auth on all protected routes,
  and React auto-escaping (no XSS). The vulnerabilities here are SUBTLE —
  the kind of mistakes experienced developers make in real production code.

  Compare:
    ShopZone    = EASY    (16+ SQLi, XSS, command injection, alg:none JWT)
    NexusBoard  = MEDIUM  (3 SQLi, stored XSS, open redirect, debug mode)
    SignalStack = HARD    (no SQLi, no XSS, no RCE — logic/config flaws only)

===============================================================================

1. IDOR — Insecure Direct Object Reference (1 endpoint only)
   - GET /api/v1/findings/:id — fetches finding by ID WITHOUT checking org membership
   - All other endpoints (assets, reports, webhooks, files) properly verify org access
   - The list endpoint GET /api/v1/findings is properly scoped
   - Only the single-resource GET bypasses the check
   - Code comment says "Organisation scoping is handled at the list level" (wrong)
   - Test: authenticate as user in Org A, request a finding ID from Org B
   - Tool: manual testing, ZAP, nuclei custom template

2. MASS ASSIGNMENT — Privilege Escalation
   - PUT /api/v1/users/profile — intended for name/bio/avatar updates
   - Code has "allowedExtras" array that includes 'role' and 'email'
   - Sending {"role": "ADMIN"} escalates privileges
   - Looks like a developer left forward-compatibility code in place
   - Test: {"name": "Test", "role": "ADMIN"}
   - Tool: manual testing, Burp Suite

3. CORS MISCONFIGURATION
   - Origin validation uses .endsWith() on stripped domain
   - Allowed origins include "signalstack.io" and "app.signalstack.io"
   - Bypass: Origin: https://evil-signalstack.io passes the check
   - credentials: true is set, so cookies are sent cross-origin
   - Null origin is NOT accepted (unlike NexusBoard)
   - Tool: manual testing, nuclei CORS templates

4. WEAK JWT SECRET
   - Secret: signalstack-jwt-2024-prod
   - Not trivial like "secret" (ShopZone) but still dictionary-crackable
   - Algorithm: HS256 only (proper — no alg:none accepted)
   - Expiry: 24h (reasonable)
   - Token properly verified against database on each request
   - Tool: jwt_tool with wordlist, hashcat

5. NO RATE LIMITING ON LOGIN
   - POST /api/v1/auth/login has no throttling
   - Proper bcrypt verification (slow by design, but still brute-forceable)
   - Error messages are generic ("Invalid email or password") — no user enumeration
   - Default credentials: admin@signalstack.io / admin123
   - Tool: hydra, custom brute-force script

6. INFORMATION DISCLOSURE — Debug Endpoint
   - GET /api/v1/debug/config — returns config, CORS settings, feature flags
   - Protected by: if (NODE_ENV === 'production') return 404
   - BUT docker-compose.yml sets NODE_ENV=development
   - Masks database password but exposes Redis URL, CORS origins, JWT expiry
   - Tool: curl, ffuf, gobuster, nuclei

7. INFORMATION DISCLOSURE — Health Endpoint
   - GET /api/v1/health — no authentication required
   - Returns: Node.js version, uptime, environment, DB status, memory usage
   - Less severe than debug/config but still leaks server fingerprinting data
   - Tool: curl, nuclei

8. INFORMATION DISCLOSURE — robots.txt
   - GET /robots.txt — discloses sensitive paths:
     Disallow: /api/v1/admin
     Disallow: /api/v1/debug
     Disallow: /internal
     Disallow: /api/v1/health
   - Tool: nikto, curl

9. WEBHOOK SSRF — Server-Side Request Forgery
   - POST /api/v1/webhooks — url field validated with isValidUrl()
   - isValidUrl() only checks: url.startsWith('http://') || url.startsWith('https://')
   - Does NOT block internal/private IPs (10.x, 172.16.x, 169.254.x, 127.x)
   - POST /api/v1/webhooks/:id/test — triggers delivery to the URL
   - Test: url=http://169.254.169.254/latest/meta-data/
   - Test: url=http://10.0.0.1:6379/ (hit Redis)
   - Requires authentication (unlike many SSRF vulns)
   - Tool: manual testing, Burp Collaborator

10. PATH TRAVERSAL — Report Template
    - GET /api/v1/reports/:id/download?template=<payload>
    - Template parameter sanitized with: .replace(/\.\.\//g, '')
    - Single-pass removal: ....// becomes ../ after replacement
    - Test: template=....//....//....//....//etc/passwd
    - Requires authentication + org membership for the report
    - Tool: manual testing, Burp Suite

11. FILE UPLOAD WEAKNESSES
    - POST /api/v1/files/upload — validates extension (.pdf, .png, .jpg, .csv)
    - Does NOT validate Content-Type header matches extension
    - sanitizeFilename() uses single-pass ../ removal (same bypass as #10)
    - Original unsanitized filename stored in DB (originalName field)
    - Requires authentication + org membership
    - Tool: Burp Suite, manual testing

12. MISSING SECURITY HEADERS
    - Present (via helmet defaults):
      X-Frame-Options: SAMEORIGIN ✓
      X-Content-Type-Options: nosniff ✓
      X-DNS-Prefetch-Control ✓
      X-Download-Options ✓
      X-Permitted-Cross-Domain-Policies ✓
      Referrer-Policy ✓
    - MISSING:
      Content-Security-Policy (CSP) — disabled in helmet config
      Strict-Transport-Security (HSTS) — disabled in helmet config
    - Code comments claim these are "handled at the reverse proxy level" (they aren't)
    - Tool: security_headers, curl

13. API TOKEN IN QUERY STRING
    - Auth middleware accepts token via ?api_key= query parameter
    - Query parameters are logged by morgan in 'combined' format
    - Tokens appear in access logs, browser history, referer headers
    - JWT tokens are valid for 24h — leaked token gives full access
    - Tool: log analysis, manual testing

14. VERBOSE ERROR MESSAGES
    - Global error handler exposes stack traces when NODE_ENV !== 'production'
    - Prisma error codes and metadata included (P2002, P2025, etc.)
    - Leaks: table names, column names, constraint names
    - Only in development mode — but docker-compose sets development
    - Tool: send malformed requests, observe error responses

15. SSL/TLS
    - Self-signed certificate (2048-bit RSA, SHA-256)
    - Better than ShopZone (which uses 1024-bit RSA, SHA-1, TLSv1.0)
    - TLSv1.2+ only, HIGH cipher suites
    - No HSTS header (see #12)
    - Tool: ssl_cert, testssl

16. COOKIE SECURITY
    - httpOnly: true ✓ (not accessible via JS — unlike ShopZone)
    - secure: false (set to true only when NODE_ENV=production)
    - sameSite: lax ✓
    - Since docker-compose sets NODE_ENV=development, secure flag is off
    - Cookie sent over HTTP connections
    - Tool: manual testing, browser dev tools

17. CLIENT-SIDE JAVASCRIPT VULNERABILITIES (Code Review)
    Found in frontend/lib/api.ts and frontend/lib/auth.ts:
    These are SUBTLE — the kind of things only a thorough code reviewer would find.

    Finding                                          | Severity  | Type
    -------------------------------------------------|-----------|-------------------
    Base64-encoded internal migration endpoint        | Medium    | Info disclosure
    in FEATURE_FLAGS._internal.migrationCallback      |           |
    Decodes to: https://internal.signalstack.io/     |           |
    __migrate/v2                                      |           |
    Base64-encoded Sentry DSN in FEATURE_FLAGS        | Low       | Info disclosure
    Decodes to: https://abc123@o.456789.ingest.      |           |
    sentry.io/4567890                                 |           |
    enableWebhookDebug flag always true               | Low       | Config flaw
    (NODE_ENV not set at build time, so              |           |
    process.env.NODE_ENV !== 'production' is true)   |           |
    Auth token in non-httpOnly cookie (SameSite=Lax) | Medium    | Insecure storage
    No Secure flag — token sent over HTTP too         |           |
    Token cached in sessionStorage                    | Medium    | Insecure storage
    (ss_auth key with token + full user object)       |           |
    Middleware checks cookie EXISTS not if valid       | Low       | Auth bypass
    (any value in signalstack_token passes)           |           |
    Open redirect via ?redirect= on /login           | Low       | Open redirect
    (middleware passes redirect param without check)  |           |

    WHY THESE ARE HARD TO FIND:
    - No obvious secrets (no API keys, no passwords in code)
    - The base64 strings are in a _internal config object that looks like build metadata
    - The Sentry DSN looks harmless but leaks org structure
    - The migration endpoint reveals internal infrastructure
    - The auth token storage issues require understanding cookie security model
    - The middleware flaw requires reading both middleware.ts and auth.ts together
    - A code reviewer needs to decode the atob() calls to find anything
